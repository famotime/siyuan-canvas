# 动态查询节点（SQL 智能卡片）使用指南 & FAQ

`siyuan-canvas` 画布支持通过 SQL 驱动的“动态智能节点”。该节点允许您在画布上直接通过 SQL 语句筛选和呈现思源块。
卡片内容是**实时生长与更新**的，配合定时刷新配置，能将您的画布打造成极具生产力的动态看板。

---

## 1. 交互入口与创建方法
您不需要使用任何新的按钮，只需在画布中按照常规的“添加笔记”交互操作即可：
1. 点击画布底部悬浮工具栏的 **“创建/关联笔记”** 按钮（或按下快捷键触发）。
2. 在弹出的搜索输入框中，输入以 **`SELECT`** 开头的 SQL 查询（大小写均可），例如：
   ```sql
   SELECT * FROM blocks WHERE content LIKE '%冥想%'
   ```
3. 下拉结果列表首行会自动识别并生成虚拟选项：`[SQL] 创建 SQL 动态查询节点`。
4. 点击该选项（或直接按下回车键），一个动态查询节点便会在画布视口中央生成。

---

## 2. 功能特性与操作

* **标题栏结果数量徽章**：卡片 Header 标题旁会实时展示当前的查询结果计数徽章。该徽章使用圆角胶囊设计，视觉清爽美观，让您在无需滚动列表时即可一目了然获取匹配数量。
* **富文本实时预览**：卡片内部展示符合条件的块列表。每个匹配项会调用思源 Markdown/HTML 解析器进行富文本渲染，确保排版（如标题级别、列表等）美观紧凑。
* **双击跳转定位**：点击列表中的任何一项，思源会立刻在主工作区中打开对应笔记，并自动**滚动、定位并高亮高能高亮**显示该原始块。
* **拖拽生长与自适应自动连线**：支持直接从列表中将某一项拖拽到画布的空白处。松开鼠标后：
  * **克隆为独立节点**：该项会被瞬间转换为画布中一个独立的思源文件卡片。
  * **中心点松手对齐**：当您在画布空白处松开鼠标时，卡片会自动调整其物理坐标，使**鼠标松手的落点精准处于新生成卡片的几何中心**，而非别扭地贴合在左上角，交互极其舒适自然。
  * **自动建立连线**：系统会自动在源 `query` 看板卡片与新生成的卡片之间**自动创建一条带箭头的指向连线**，建立清晰的来源引用关系。
  * **方位自适应锚定**：系统会根据您松开鼠标的相对方向（上下左右），自动将连线的端点锚定在最合理的侧边（例如：将新卡片拖到右侧松开，连线将自动从源的 `right` 端点连到新卡片的 `left` 端点）。
  * **一步撤销**：整个拖出及连线动作采用原子事务提交，您只需按下一次 `Ctrl + Z` 即可撤销整个拖拽行为（连线和新卡片会同时消失）。
  > [!TIP]
  > **操作锁定**：我们在列表项上实现了防事件冒泡机制。当您在列表内按下并拖拽某一项时，**动态卡片本身会被锁死在原地，绝不会发生位置上的跟随移动**。
* **编辑与配置**：双击卡片、或点击卡片 Header 上的“编辑”图标即可展开编辑。您可以修改 SQL 语句、设置自动刷新间隔、或限制最大结果展示数。

## 3. 看板最佳实践（防“数据淹没”指南）

在思源底层的 `blocks` 数据表中，文档本身也是一种特殊的块（`type = 'd'`）。但文档块的 `content` 字段通常为空（或只存放标题路径），文档正文文字全部存放在子级的内容块（段落、列表项等）中。

如果您直接使用 `content LIKE '%关键词%'` 查询：
1. **搜不到文档本身**：因为文档块的 content 为空，只能搜到正文里提到的碎片块。
2. **列表容易被碎块淹没**：如果笔记中多次提到该关键词，成百上千的内容碎块会将真正的文档推到数百名之后，进而被列表的最大渲染上限（默认 50 条）截断过滤。

### 💡 推荐的黄金 SQL 看板写法
为了同时查到**标题含关键词的“文档”**与**正文含关键词的“块”**，并让“文档”永远优先置顶，强烈推荐配合 `ORDER BY` 使用以下查询：
```sql
SELECT id, type, content, hpath, markdown FROM blocks 
WHERE content LIKE '%冥想%' OR (type = 'd' AND hpath LIKE '%冥想%')
ORDER BY (case when type = 'd' then 0 else 1 end) ASC
```

## 4. FAQ 常见问题解答

### Q1：为什么我查到了“冥想”的内容，但是结果项全被标注为“块”，没有一个被标注为“文档”？
**A**：有两个主要原因：
1. **SQL 漏掉了 `type` 字段**：若您写了如 `SELECT id, content FROM blocks` 这样的局部列查询，导致前端无法读取到类型属性。请使用 `SELECT *` 或确保查询列包含 `type`（如 `SELECT id, type, content, hpath`）。
2. **文档 `content` 字段为空**：思源中文档块（`type='d'`）的标题实际存放在 `hpath` 或 `path` 中，其 `content` 字段为空。单纯的 `content LIKE` 无法匹配到文档块本身。您需要使用上面推荐的 `OR (type = 'd' AND hpath LIKE '%关键词%')` 进行联合查询。

### Q2：我设置了“最大展示数量 500 条”，为什么不加排序时只能在第 61 项看到一个文档，其他文档都消失了？
**A**：
没有排序时，数据库会根据底层的物理物理存储顺序输出。如果您库里有上百个包含该词的普通内容块，它们会把文档块“挤”到 500 条甚至更靠后的位置，被渲染上限丢弃。
加上 `ORDER BY (case when type = 'd' then 0 else 1 end) ASC` 能够强制让类型为 `"d"`（即文档）的权重置顶（值为0），确保它们一定排在最前列展示。

### Q3：为什么强烈建议不要在动态节点中使用 `SELECT *`？
**A**：
文档块（`type = 'd'`）在 `markdown` 和 `dom` 字段中存放着**整篇笔记的所有数据**。`SELECT *` 会强行让思源提取这些庞大的文本，极易发生传输过载，导致卡片刷新缓慢。
因为看板只需要摘要，最合理且高效的法则是只检索核心列：
```sql
SELECT id, type, content, hpath, markdown FROM blocks ...
```

### Q4：定时自动刷新会降低思源的运行效率吗？
**A**：
自动刷新会周期性地发起 SQL 查询。为保证思源流畅性：
1. 当您处于编辑卡片状态时，定时器会安全暂停，不会打断您的输入。
2. 当卡片被删除或画布关闭时，定时器会自动注销并彻底回收。
3. **建议日常将刷新间隔设为 0**（关闭自动刷新，采用手动点击刷新按钮）。只在需要将画布投放到大屏作为大盘看板时，再通过属性面板设置定时（如设为 30 或 60 秒）。
